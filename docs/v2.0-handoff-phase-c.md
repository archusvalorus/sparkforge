### How Boss Mode works now

- **Entry:** title → ☠ BOSS MODE. The roster lists every boss you have
  **actually felled** — `BossEntry.isUnlocked` reads `hasDefeatedBoss(id)`, so
  the mode appears as soon as *any* boss is down and grows as you fell more.
  It is **not** gated behind Arena 5. This is good design; leave it alone.
- **Opener:** 8 granted LEVELS, either **DRAFT** (you pick all 8 cards in
  sequence) or **RANDOM** (ARAM chaos). Draft is the default and makes Boss
  Mode a build testbed; random stays one tap away.
- **Order:** SEQUENTIAL or MIXUP (same set, shuffled — draws without
  replacement, so no boss repeats and run length is identical).
- **Per stage:** each boss in its own home arena · no adds · 2–3 randomized HP
  orbs between bosses · arena bosses ramp `+12.5% × stage`, monuments flat
  `+25%`.
- **Death is final.** No ads, no revives. Retry starts a fresh gauntlet.
- **Rewards:** sandbox-isolated. The 100 Forge XP cap is the **full-clear**
  reward; a run earns the share it cleared (1 of 4 = 25). Felling nothing pays
  a flat 5 consolation.

### Hard-won lessons worth not relearning

- **Starting level, not the XP multiplier, is the lever.** Boss 1 was paying
  ~10 levels because levels 2–6 cost 43 XP combined — under a fifth of one
  boss reward. Halving the multiplier still paid 8 levels. Granting levels
  (`PlayerNode.grantLevel()`) is what fixed it. Any future mode that starts a
  player at level 1 with a big lump-sum XP source will hit this same wall.
- **SKActions do not care about `gameState`.** A scheduled stage handoff fired
  behind a level-up modal and was silently dropped, deadlocking the whole mode.
  Distinguish *terminal* states (`.dead`) from *transient* ones (`.levelUp`,
  `.paused`, `.synergyReveal`): terminal cancels, transient **defers**.
- **A bool is not a gradient.** `forgeXPForRun(bossDefeated:)` takes a Bool, so
  felling one boss awarded its flat +100 — every gauntlet paid exactly the cap
  regardless of depth. Check that a "cap" is not silently a constant.
- **A debug seam that can be left on must announce itself.** A stale build with
  `debugStartStage = 5` cost a confusing playtest. The Boss Mode modal now
  shows an amber `⚠︎ DEBUG` line whenever a seam is hot. Follow this pattern
  for any new dev flag.

### Dev seams (DEBUG-only, ship cold)

`GameConfig.BossMode.debugForceFullRoster` (offer every boss regardless of
defeat records) and `debugStartStage` (open the gauntlet at stage N). Both are
`#if DEBUG` and ship at `false` / `1`. **Verify them cold at HEAD before every
push** — and reinstall the app on the simulator after reverting them, or you
will be testing a stale build.

## NEXT: Phase C

Three pieces, all designed, none started. Suggested order below — the Growth
tree is the big one and the skins are the shippable-in-a-day one.

### C1 — The Growth tree (the 7th tag)
Full creative handoff already exists: **`docs/growth-tree-creative-handoff-v2.0.md`**.
Terra gates the pool, Tree is the capstone, the verb is **Cultivate**, and
Growth modifies the **arena** rather than Spark. The key insight from that doc:
**the v1.9 capstones ARE the Growth engine** (zones, priority-target turrets,
the tier→event ladder), so this is more assembly than invention.
See the `plant-growth-skill-tree` memory. Its dark sibling, **Decay**, is
deliberately reserved for post-v2.0 — do not build rot/infection into Growth.

### C2 — The Panda secret tree
The `panda-secret-tree` memory has the whole design: masked `???` forever, a
9.27% run mutation, deterministic once taken, escalating to a flaming panda
kaiju plus a 5% Panda Samurai that deletes one priority target. Buildable
almost entirely from banked systems (FamiliarNode, morph, priority-target,
execute, the tier engine, the conditional pool).

### C3 — Premium + Panda skins
The earned/premium framework shipped in Unit 1, so this is content, not
engineering. See `skin-system-canon`: $4.99 baseline / $9.99 special, roster =
earned Spark, premium Spark, earned Panda, premium Panda (flaming kaiju).
