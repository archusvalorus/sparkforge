# v1.9 Unit 1 — Card-Leveling Engine: Architecture Plan

*Drafted July 15, 2026 (end of the v1.8 ship session, while fully oriented to
the card system) so a fresh session executes from an approved plan. This is the
load-bearing keystone of the whole roadmap — the goal is a backward-compatible
engine that touches as little as possible and changes ZERO existing card
definitions. See [[card-leveling-system]] and `docs/v1.9-kickoff.md` (Unit 1).*

## Goal / DoD (from the kickoff)
Owned, non-maxed cards re-appear in draws and **level up** instead of being
excluded. **1-tier cards behave exactly as v1.8** (regression-free). Tier +
next-tier are legible on the card. **Per-run only — no save-data change.**

## The current system (what we're changing)
- `UpgradeManager.UpgradeCard` = `{ id, name, tag, secondaryTag?, description,
  apply: (PlayerStats) -> Void }`. `apply` mutates additive `PlayerStats` fields.
- `buildCardPool()` builds ~68 cards into one array.
- `pickedCardIDs: [String]` — each card picked at most once.
- `drawCards(count:)` — filters `!pickedCardIDs.contains(id)`, then tag-diverse
  (v1.6: each drawn card from a different tree).
- `pickCard(card)` — records id, increments `tagCounts` (primary + secondary,
  excluding `.neutral`); `checkSynergies` fires at tag counts 3/5/7.

## The model change — additive, zero churn on existing cards

**Recommended (Fork A → Option B, additive):** keep `apply` as tier 1; add an
optional ladder of *higher* tiers. Existing cards are untouched and implicitly
1-tier.

```swift
struct UpgradeCard {
    let id: String
    let name: String
    let tag: Tag
    var secondaryTag: Tag? = nil
    let description: String
    let apply: (PlayerStats) -> Void          // tier 1 (unchanged)
    // v1.9: tiers 2..N. Each closure is the DELTA applied on that level-up.
    var higherTiers: [(PlayerStats) -> Void] = []
    // v1.9: per-tier copy for the selection/Codex UI. nil → use `description`.
    var tierDescriptions: [String]? = nil
}
extension UpgradeCard {
    var maxTier: Int { 1 + higherTiers.count }
}
```
- A card with no `higherTiers` = **maxTier 1 = identical to today**.
- Laddered cards (Unit 3) add `higherTiers: [tier2Delta, tier3Delta, ...]` and
  `tierDescriptions`. Rungs are authored as **deltas** (`+= …`) or new-flag sets
  for capstones — NOT re-sets of the whole effect.
- *Rejected (Option A, unified `tiers: [closure]` array):* cleaner long-term but
  forces wrapping all ~68 cards now. Save it as an optional later cleanup; the
  engine unit should not touch existing card bodies.

## State — per-run tier tracking
Replace the single-pick model with a tier map (still per-run, reset each run —
no persistence, like `pickedCardIDs` today):
```swift
private(set) var cardTiers: [String: Int] = [:]   // id → current tier (1…maxTier)
```
- `pickedCards` (pause build viewer) derives from `cardTiers.keys`.
- Keep `pickedCardIDs` as a derived convenience if other code reads it, or
  migrate readers to `cardTiers.keys` (grep for `pickedCardIDs` consumers).
- `reset()` clears `cardTiers` (mirrors current reset of `pickedCardIDs`).

## drawCards — re-offer non-maxed cards
Change eligibility from "never picked" to "not maxed":
```swift
let available = allCards.filter { (cardTiers[$0.id] ?? 0) < $0.maxTier }
```
Keep the v1.6 tag-diverse shuffle/selection **unchanged** — owned non-maxed
cards flow through the same logic as new cards.
- **Fork B (draw weighting):** simplest = treat owned-non-maxed identically to
  new cards (recommended — ship it, tune in playtest). Do NOT add level-up-vs-
  new weighting in the engine unit; that's a tuning knob for later if draws feel
  off.

## pickCard — run the next rung, keep tags orthogonal
```swift
func pickCard(_ card: UpgradeCard) {
    let cur = cardTiers[card.id] ?? 0          // 0 = not owned
    let next = cur + 1
    if cur == 0 { card.apply(stats) }          // tier 1
    else { card.higherTiers[cur - 1](stats) }  // tier 2..N delta
    cardTiers[card.id] = next
    if cur == 0 {                              // tag counts ONLY on first pick
        if card.tag != .neutral { tagCounts[card.tag, default: 0] += 1 }
        if let s = card.secondaryTag, s != .neutral { tagCounts[s, default: 0] += 1 }
        // …existing checkSynergies / build-hint logic, first-pick only
    }
}
```
- **This is the orthogonality guarantee:** leveling a card does NOT advance tag
  counts, so synergies stay a pure *breadth* axis. Locked decision — don't let
  tiers feed synergies.
- Verify `stats` handle: `pickCard` needs the live `PlayerStats`. Today the
  apply closure is invoked from the level-up flow — confirm where (GameScene's
  card-selection handler) and route `higherTiers` through the same path.

## UI — selection card shows tier + next effect
In the card-selection node (`UpgradeCardNode` / the level-up selection in
`GameScene.showCardSelection`):
- If `cur == 0`: render as today (new card, its `description`).
- If `cur >= 1`: show a **tier badge** (`Lv {cur} → {cur+1}`) and the
  next-tier copy (`tierDescriptions[cur]` if present, else `description`), plus a
  subtle "LEVEL UP" affordance so owned cards read distinct from new ones.
- Pause **build viewer** (`CardDetailNode`): show each owned card's current tier.
  Engine exposes the data; full Codex ladder render is **Unit 2**.
- Keep it minimal here — polish + per-tier copy land with signature ladders
  (Unit 3). **Fork C:** how rich the tier UI is now — recommended MINIMAL
  (badge + next-effect line), no ladder visualization until Unit 2/3.

## Test strategy (validate the engine end-to-end)
Signature ladders are Unit 3, but the engine needs ONE laddered card to prove
it works. Temporarily give an existing card (e.g. the multishot/scatter) a
2–3 tier `higherTiers` ladder behind a DEBUG flag or as a throwaway, verify:
1. First pick applies tier 1; the card re-appears in later draws.
2. Re-picking levels it (tier badge increments, delta applies, stats grow).
3. At maxTier it stops re-appearing (excluded).
4. Tag count incremented once (first pick only) — synergies unaffected by levels.
Then remove/park the temp ladder for Unit 3 to author for real.

## Regression checklist (the backward-compat contract)
- Every existing (1-tier) card: picked once → maxed → never re-offered (exactly
  v1.8 behavior).
- `tagCounts` / synergy tiers fire identically to v1.8 for 1-tier cards.
- `reset()` clears tier state between runs; no UserDefaults writes added.
- Build viewer + any `pickedCardIDs` consumers still work.

## Touch points (files)
- `Systems/UpgradeManager.swift` — struct, `cardTiers` state, `drawCards`,
  `pickCard`, `reset`, `pickedCards`.
- `Scenes/GameScene.swift` — `showCardSelection` / the card-pick handler (route
  higher-tier apply; pass tier to the card UI).
- `Nodes/UpgradeCardNode.swift` — tier badge + next-tier copy + level-up affordance.
- `Nodes/CardDetailNode.swift` — show current tier in the build viewer (light).
